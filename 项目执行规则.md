# 番茄五书项目执行规则

> 本文件是整个仓库的通用执行规则，适用于原作研究、数据整理、质量校验、文档维护和 Git 交付。
>
> 规则目标：保持原作事实和分析结论可追溯、保持长任务可断点续作，避免任务偏移、事实漂移和产物失真。

---

## 读取策略

项目采用四级读取，默认只到当前任务所需的最低级别：

0. **导航**：只读 `AGENTS.md` 和 `docs/当前接手卡.md`；不打开业务资料，不做任何全库判断。
1. **定位**：只读相关规则条款、目标文件和必要的章节定位；默认预算为 1 个规则文件、1 个索引、1 个大纲/state、2 个原文分章文件。
2. **证据**：只有事实不清、需要连续性或出现冲突时，才增加相邻章节、相邻 state 或目标汇总条目。
3. **专项核查**：只有用户明确要求或任务必须依赖时，才检查目录、章节数量、分章完整性、全量附录、CSV 或更大范围原文。

扩大读取范围前必须说明触发原因；没有矛盾就停止升级。规则文件是执行手册，不是接手资料。原作规则、子 Agent 规则、质量闸门和 Git 规则均按触发条件读取，不要求通读。大文件优先使用 `tools/query_material.py` 定点查询。

## 一、项目范围与事实边界

本项目当前正式范围为五部原作研究：

- 《吞噬星空》
- 《莽荒纪》
- 《飞剑问道》
- 《沧元图》
- 《雪鹰领主》

正式产物包括五部原作目录中的索引、分章、逐篇/逐卷/逐集大纲、`_state/` 状态档案，以及五书全量附录、特征数据和横向节奏分析。所有分析结论必须能回到原文、索引或已落盘 state，不得把推断伪装成原文事实。

---

## 二、规则优先级

出现冲突时，按以下顺序判断：

1. 用户当前明确指令；
2. 本文件的项目级规则；
3. 原作分析规则 `原作分析规则.md`；
4. 当前执行版总纲、索引、state 和已验收产物；
5. 历史文档、旧版本、临时分析和 Agent 建议。

历史材料只能作为参考，不能覆盖当前执行版本。任何 Agent 输出都只是候选素材，未经主创上下文核对，不得自动成为正式设定。

---

## 三、任务启动规则：先定边界，再开始工作

每个中型或大型修改任务开始前，内部形成一张简短任务卡：

- 目标：这次要解决什么问题；
- 范围：允许读取和修改哪些文件；
- 非目标：明确哪些事情本次不做；
- 依赖：完成本次修改前实际需要读取哪些总纲、索引、state 或研究材料；不得把“可能有用”当成依赖；
- 产物：最终写入哪些文件；
- 验收：用什么数量、结构、事实或一致性标准判断完成；
- 风险：哪些情况必须暂停并上报。

不得因为发现相关问题就自动扩大任务范围。若需要改变卷数、境界体系、主题、主角弧光、结局或核心创作方向，必须先向用户确认。

开始修改前检查当前 Git 状态，保留用户已有修改，不覆盖、不回滚、不删除未确认的内容。仅了解项目时不要求先做完整 Git 或文件结构审计。

任务卡至少包含“目标、只读范围、禁止读取/修改范围、产物、验收、存疑”。任务结束时只向任务日志当前摘要或本次任务记录落盘“完成、未完成、存疑、下一步目标文件”，不把完整过程复制进任务日志顶部；静态接手卡不记录临时任务。

## 四、研究产物维护规则

### 4.1 大纲必须回到原文

逐章大纲应直接依据单章原文和必要的相邻章节修订。发现旧大纲仍保留占位、推断、旧分割口径或过期 state 说明时，应优先回查当前 `resources/` 分章文件，再更新对应章节条目和 state。

### 4.2 差异说明必须具体

原文、索引、分章文件、大纲和 CSV 之间存在差异时，说明应写清：

- 当前原文或分章文件的实际情况；
- 大纲采用的分析口径；
- 是否需要后续脚本或索引修正。

不要求为了表面统一而制作额外章节口径总表；只在目标文件里修正会影响接手判断的条目。

### 4.3 汇总产物跟随章节大纲

五书全量附录、特征数据和横向分析属于二级汇总产物。若章节大纲发生事实性改动，后续维护时应同步检查对应汇总条目是否仍含旧占位或旧存疑说明。

## 五、原作研究规则

原作研究必须忠于原文：

1. 情节、人物、设定、数据、时间、境界和因果必须能追溯到原文或明确索引；
2. 不确定内容先查单章原文，再查相邻章节和 state；
3. 仍无法确认时标记 `【存疑】`，不得猜测补全；
4. 章节不得擅自合并、拆分、跳过或凑数；
5. 索引、分章文件、大纲和 CSV 发生差异时，保留差异并记录原因，不得静默改成“看起来统一”；
6. 原文存在错号、重复、缺章或抓取占位符时，分别记录“原文异常”和“分析口径”，不能把修正后的口径伪装成原文事实。

所有五部原作的逐章分析统一遵守 `原作分析规则.md`，不再设置任何单独作品的规则文件。

## 六、连续性与状态管理

长篇逐章分析必须维护以下连续性：

- 人物当前状态、境界、伤势、关系和动机；
- 时间线、年龄、事件先后和时间跨度；
- 地图、势力、组织关系和资源分布；
- 能力、装备、法门、限制和代价；
- 主线、支线、情感线和家族线；
- 伏笔状态：未埋、已埋、推进中、已回收、废弃及废弃原因；
- 每篇/卷/集目标、阶段结果和卷尾钩子。

每完成一个阶段或一篇/卷/集，必须更新对应 state 或主产物，并做一次跨章、跨卷检查。不能只依赖聊天上下文记忆。

项目阶段、待办或交接重点发生变化时，必须同步更新 `docs/任务日志.md` 顶部的“当前状态摘要”；历史日志只保留追溯用途。

## 七、并行与子 Agent 规则

只有在任务相互独立、读取边界清晰、写入文件不冲突且验收标准明确时，才允许并行。

必须顺序执行的工作包括：

- 依赖前文状态的跨篇/跨卷/跨集校对；
- 同一文件或同一设定表的修改；
- 会改变索引、分章、大纲和汇总产物对应关系的修订。

使用子 Agent 时必须提供：目标、边界、必读文件、写入权限、唯一产物、验收标准、存疑处理方式和最终回报要求。子 Agent 失联时必须探活、接管或重派，不得无限等待或向用户声称正常运行。

子 Agent 回报只保留：结论、修改文件/章节、证据文件、存疑、未完成项和下一步建议；禁止复制整段原文、完整大纲、完整 CSV 或无关过程日志。

## 八、质量闸门

### 分析或修订开始后

- 已读取本次修改真正需要的目标原文、索引、相关大纲或 state；
- 只确认与本次修改有关的人物、境界、时间线和篇/卷/集目标；
- 已明确本次修订内容的证据来源和边界。

上述检查不属于项目接手阶段，也不要求为了解项目而提前验证全库。

### 阶段完成后

- 章节/卷数量与索引或已记录差异说明一致；
- 每章字段完整，章节没有漏写或重复；
- 人物状态和境界前后连贯；
- 伏笔有登记，未回收项有后续安排；
- 章末钩子和伏笔记录与原文一致；
- 篇/卷/集末结果兑现本篇/卷/集主线，并留下合理的下一阶段入口；
- 发现的问题已写入任务日志或存疑清单。

### 交付前

- 本次涉及的产物文件真实存在且非空；
- 本次声明需要的自动或人工校验已经执行；
- 本次涉及的 README、任务日志和实际状态没有明显冲突；
- Git 差异只包含本次任务范围；
- 用户要求“提交”时，默认执行“新建分支、commit、push 分支、合并回 `main`、push `main`”的完整交付流程，并确认两条远程分支均已同步。

### 状态读取

- state 先取快速视图：人物当前状态、核心主线、近章摘要、待确认项；
- 只有跨篇连续性、历史伏笔或冲突核查才读取完整 state；
- 不复制 `state_quick.md` 和完整 state 两套长期文件，快速视图由查询工具按标题即时提取；
- 单个 state 超过约 180 行时，优先进一步拆分或压缩，再继续扩大读取范围。

## 九、必须暂停并上报的情况

遇到以下情况不得擅自解决：

- 索引、原文、大纲或 state 互相矛盾；
- 需要改变已验收的大范围分析口径；
- 需要把历史方案重新启用；
- 需要合并或拆分章节；
- 无法确认某个事实、因果或人物动机；
- 发现前文错误会导致大范围返工；
- 任务范围需要修改用户未授权的外部系统或远程数据。

统一回报格式：

```text
【中断】位置：……
问题：……
已核对：……
影响：……
需要用户决定：……
```

## 十、版本与交接规则

长任务必须阶段性落盘，记录完成内容、存疑、待办和下一步。禁止只把关键决策留在聊天上下文中。

Git 交付约定：

1. 开始修改前检查 `git status`、当前分支和远程同步状态；
2. 用户说“提交项目”“提交并推送”或“提交”时，即视为已经授权完整 Git 交付，不重复确认操作意图；必须从当前基线新建本次提交专用分支，不得直接在 `main` 上提交；
3. 分支名统一使用 `codex/YYYYMMDD-XX`，其中日期使用本地日期，`XX` 是当天从 `01` 开始递增的提交序号；
4. 只提交属于本次任务的改动，不覆盖、不回滚、不混入用户无关修改；
5. 提交前必须更新 `CHANGELOG.md`，记录日期、分支、主要修改、验证结果和推送状态；
6. 提交前必须更新 `README.md`，确保目录结构、当前产物、归档位置和使用边界与实际仓库一致；
7. 提交信息必须概括本次修改的主要内容，不使用“update”“修改文件”等无信息量描述；
8. 提交后必须先推送当前分支到 `origin`，不得只执行本地 commit；
9. 当前分支推送成功后，必须切换到 `main`，将本次分支合并回 `main`，再推送 `main` 到 `origin`；
10. 合并完成后确认 `main` 与 `origin/main` 一致，并向用户明确报告工作分支、提交号、合并结果和推送结果；
11. 不使用破坏性回滚命令覆盖用户工作。

用户的“提交”默认包含：本地 commit、推送工作分支、合并回 `main`、推送 `main` 和确认远程同步。只有认证、网络、权限或工具失败时，才停止并报告实际失败原因，不把正常交付流程再次变成确认问题。

### 提交前固定检查清单

- `git status` 无未说明的无关改动；
- `README.md` 的目录树与实际文件、归档路径一致；
- `CHANGELOG.md` 已新增本次变更记录；
- `git diff --check` 通过；
- 提交信息包含主要修改内容；
- 当前工作分支已推送；
- 当前工作分支已合并回 `main`；
- `main` 已推送且与 `origin/main` 同步。
