# 《番茄五书与〈界潮〉》项目执行规则

> 本文件是整个仓库的通用执行规则，适用于原作研究、数据整理、原创小说《界潮》策划、质量校验、文档维护和 Git 交付。
>
> 规则目标：保持创作方向稳定、保持事实和设定可追溯、保持长任务可断点续作，避免任务偏移、设定漂移和产物失真。

---

## 一、项目分层与事实边界

本项目必须始终区分两类工作：

1. **原作研究层**：对五部原作进行索引、分章、逐章概括、标签提取和节奏分析。
2. **原创创作层**：以研究结论为参考，创作《界潮》的世界观、人物、总纲、卷纲和逐章规划。

原作研究层产出的是“证据”；《界潮》产出的是“创作决策”。不得把原创规划伪装成原作事实，也不得把原作人物、事件、台词或名场面直接移植到《界潮》。

当前《界潮》唯一执行版本：

- 22卷；
- 990章；
- 每章4000—5000字，默认4500字；
- 原36卷、1620章方案已废止，不属于当前项目资料和执行依据；不得引用、恢复或续写。

当前创作权威文件：

1. `原创小说_界潮修订总纲框架.md`：不可变规格和前期节奏约束；
2. `原创小说_界潮总纲.md`：当前总纲、世界观、人物、阶段和卷级结构；
3. `原创小说_界潮逐章详细总纲_第一阶段.md`：已经确认的第一卷逐章执行样稿；
4. `原创小说_界潮六书前100章横向节奏对比表.md`：研究参考，不是《界潮》正文证据。

---

## 二、规则优先级

出现冲突时，按以下顺序判断：

1. 用户当前明确指令；
2. 本文件的项目级规则；
3. 原作分析规则 `原作分析规则.md`；
4. 当前执行版总纲、索引、state 和已验收产物；
5. 历史文档、旧版本、临时分析和 Agent 建议。

历史材料只能作为参考，不能覆盖当前执行版本。任何 Agent 输出都只是候选素材，未经主创上下文核对，不得自动成为正式设定。

---

## 三、任务启动规则：先定边界，再开始工作

每个中型或大型任务开始前，必须明确：

- 目标：这次要解决什么问题；
- 范围：允许读取和修改哪些文件；
- 非目标：明确哪些事情本次不做；
- 依赖：必须先读取哪些总纲、索引、state 或研究材料；
- 产物：最终写入哪些文件；
- 验收：用什么数量、结构、事实或一致性标准判断完成；
- 风险：哪些情况必须暂停并上报。

不得因为发现相关问题就自动扩大任务范围。若需要改变卷数、境界体系、主题、主角弧光、结局或核心创作方向，必须先向用户确认。

开始前必须检查当前 Git 状态，保留用户已有修改，不覆盖、不回滚、不删除未确认的内容。

## 四、原创创作规则

### 4.1 核心创作必须保持单一主创视角

以下内容必须由同一个主创上下文顺序决定并统一写入主产物：

- 主题和核心命题；
- 主角、核心配角和反派的人物弧光；
- 世界观边界、力量体系和地图升级；
- 全书卷数、章数、字数和阶段功能；
- 长线伏笔、误导、推进与回收；
- 情感线、家族线、师门线和战争线；
- 终局问题、最终大战和结局兑现。

子 Agent 可以做资料整理、独立核验、统计分析和局部只读审查，但不得分别创作核心总纲后直接拼接。

### 4.2 每章必须有可验收功能

《界潮》逐章规划至少记录：

- 本章剧情结果；
- 人物变化；
- 能力、资源或身份变化；
- 冲突或任务推进；
- 爽点、情绪点或代价；
- 伏笔埋设、推进或回收；
- 章末钩子；
- 与本卷目标及全书主线的关系。

不能只写“发生了什么”，还必须说明“本章完成了什么结构功能”。

### 4.3 设定变化必须有代价和影响

任何新增或修改的能力、境界、势力、地图、宝物、规则或人物关系，都必须同步检查：

- 是否违反已锁定边界；
- 是否改变战力平衡；
- 是否影响此前伏笔；
- 是否改变人物选择和责任范围；
- 是否需要同步修改后续卷纲。

不能为了制造爽点临时增加无代价能力，不能用新设定覆盖旧矛盾而不记录修订原因。

## 五、原作研究规则

原作研究必须忠于原文：

1. 情节、人物、设定、数据、时间、境界和因果必须能追溯到原文或明确索引；
2. 不确定内容先查单章原文，再查相邻章节和 state；
3. 仍无法确认时标记 `【存疑】`，不得猜测补全；
4. 章节不得擅自合并、拆分、跳过或凑数；
5. 索引、分章文件、大纲和 CSV 发生差异时，保留差异并记录原因，不得静默改成“看起来统一”；
6. 原文存在错号、重复、缺章或抓取占位符时，分别记录“原文异常”和“分析口径”，不能把修正后的口径伪装成原文事实。

所有五部原作的逐章分析统一遵守 `原作分析规则.md`，不再设置任何单独作品的规则文件。

---

## 六、连续性与状态管理

长篇创作必须维护以下连续性：

- 人物当前状态、境界、伤势、关系和动机；
- 时间线、年龄、事件先后和时间跨度；
- 地图、势力、组织关系和资源分布；
- 能力、装备、法门、限制和代价；
- 主线、支线、情感线和家族线；
- 伏笔状态：未埋、已埋、推进中、已回收、废弃及废弃原因；
- 每卷目标、阶段结果和卷尾钩子。

每完成一个阶段或一卷，必须更新对应 state 或主产物，并做一次跨章、跨卷检查。不能只依赖聊天上下文记忆。

---

## 七、并行与子 Agent 规则

只有在任务相互独立、读取边界清晰、写入文件不冲突且验收标准明确时，才允许并行。

必须顺序执行的工作包括：

- 核心世界观决策；
- 主角和主要人物弧光；
- 总纲、卷纲和终局；
- 依赖前卷结论的后卷创作；
- 同一文件或同一设定表的修改。

使用子 Agent 时必须提供：目标、边界、必读文件、写入权限、唯一产物、验收标准、存疑处理方式和最终回报要求。子 Agent 失联时必须探活、接管或重派，不得无限等待或向用户声称正常运行。

---

## 八、质量闸门

### 写作前

- 已读取当前总纲和相关 state；
- 已确认本任务不使用历史废止方案；
- 已确认人物、境界、时间线和卷级目标；
- 已明确本次新增内容的功能和边界。

### 阶段完成后

- 章节/卷数量与计划一致；
- 每章字段完整，章节没有漏写或重复；
- 人物状态和境界前后连贯；
- 伏笔有登记，未回收项有后续安排；
- 章末钩子能推动下一章；
- 卷末结果兑现本卷承诺，并留下合理的下一阶段入口；
- 发现的问题已写入任务日志或存疑清单。

### 交付前

- 产物文件真实存在且非空；
- 自动或人工校验已经执行；
- README、任务日志和实际状态没有明显冲突；
- Git 差异只包含本次任务范围；
- 用户要求“提交”时，默认执行 `commit + push`，并确认远程分支已同步。

## 九、必须暂停并上报的情况

遇到以下情况不得擅自解决：

- 总纲、索引、原文或 state 互相矛盾；
- 需要改变当前 22 卷、990 章执行版本；
- 需要更换主角、主题、结局或核心世界观；
- 需要把历史方案重新启用；
- 需要合并或拆分章节；
- 无法确认某个事实、因果或人物动机；
- 发现前文错误会导致大范围返工；
- 任务范围需要修改用户未授权的外部系统或远程数据。

统一回报格式：

```text
【中断】位置：……
问题：……
已核对：……
影响：……
需要用户决定：……
```

## 十、版本与交接规则

长任务必须阶段性落盘，记录完成内容、存疑、待办和下一步。禁止只把关键决策留在聊天上下文中。

Git 交付约定：

1. 开始修改前检查 `git status`、当前分支和远程同步状态；
2. 用户要求“提交项目”时，必须从当前基线新建本次提交专用分支，不得直接在 `main` 上提交；
3. 分支名统一使用 `codex/YYYYMMDD-XX`，其中日期使用本地日期，`XX` 是当天从 `01` 开始递增的提交序号；
4. 只提交属于本次任务的改动，不覆盖、不回滚、不混入用户无关修改；
5. 提交前必须更新 `CHANGELOG.md`，记录日期、分支、主要修改、验证结果和推送状态；
6. 提交前必须更新 `README.md`，确保目录结构、当前产物、归档位置和使用边界与实际仓库一致；
7. 提交信息必须概括本次修改的主要内容，不使用“update”“修改文件”等无信息量描述；
8. 提交后必须推送当前分支到 `origin`，不得只执行本地 commit；
9. 推送后确认当前分支与远程分支一致，并向用户明确报告分支名、提交号和推送结果；
10. 不使用破坏性回滚命令覆盖用户工作。

### 提交前固定检查清单

- `git status` 无未说明的无关改动；
- `README.md` 的目录树与实际文件、归档路径一致；
- `CHANGELOG.md` 已新增本次变更记录；
- `git diff --check` 通过；
- 提交信息包含主要修改内容；
- 当前分支已推送且远程同步。
