# 项目协作入口

本文件只负责路由任务，不要求新 Agent 在接手阶段验证项目内容。

## 首次接手默认协议（必须先执行）

当用户首次提供仓库地址，或使用“了解项目”“分析项目”“看看项目是什么情况”“仔细看看”“了解项目背景”等表达时，**一律先按“首次接手概览”处理**。这些表达不等于审计、验证或问题排查；用户不需要记住或指定模式名称。

### 首次接手概览允许做的事

1. 读取本文件和 `docs/当前接手卡.md`；
2. 只查看仓库根目录和最多两层的目录/文件名；
3. 如果项目用途仍不清楚，只读取 `README.md` 的开头摘要，以及最多 3 个小型配置文件或明显入口文件；
4. 用简短文字说明项目用途、直接可见的技术栈、主要目录和工作流程；无法从允许范围确认的内容标记为“待进一步读取”；
5. 概览完成后停止，等待用户指定下一步。

### 首次接手概览禁止做的事

- 不扫描全库或递归读取业务文件内容；
- 不统计文件、章节、字数、数据行或产物数量；
- 不验证章节数量、文件完整性、重复、缺失、CSV 覆盖率、内容正确性、运行状态或历史问题；
- 不运行测试、分割脚本、批量分析、校验脚本或其他全库命令；
- 不打开整部原文、完整大纲/state、全量附录、CSV 或完整任务日志；
- 不为了“仔细”“全面”“深入”自行扩大读取范围；
- 不创建子 Agent 来替代上述禁止操作；
- 不把 README、历史日志或已有“完成/已修复”声明当作本次验证结论。

**“默认不验证”不等于“可以自行验证”。未经用户明确授权，验证、统计、扫描、审计和问题排查均禁止。**

只有用户明确提出“审计、核查、验证、统计、检查完整性、找缺失、检查重复、找问题”等同等含义的要求，或具体修改确实依赖某项事实核对时，才可进入更高读取级别；扩大前必须说明原因和范围。普通概览不得主动升级。

## 接手顺序

1. 先读本文件；
2. 再读 `docs/当前接手卡.md`；
3. 若属于首次接手概览，按上面的允许范围完成后停止；
4. 只有用户明确要求下一步工作、实际修改或明确核查时，才定位目标文件并读取原文、索引、state、目录和汇总数据。

项目概览阶段不检查大纲正确性、章节数量、文件结构完整性、历史问题或全库状态。README 和任务日志是导航/历史资料，不是接手阶段必读资料；其中的“完成”“已修复”等文字也只是路由信息，不能替代本次核查。

## 四级读取与默认预算

- **0级：导航**：只看本文件和接手卡，不打开业务资料。
- **1级：定位**：读取目标规则条款、目标文件局部和目标章节；默认不超过 1 个规则文件、1 个索引、1 个大纲/state、2 个原文分章文件。
- **2级：证据**：仅因事实不清或需要连续性，再增加相邻章节、相邻 state 或目标汇总条目。
- **3级：专项核查**：只有用户明确要求或任务必须依赖时，才检查目录、章节数量、分章完整性、全量附录、CSV 或更大范围原文。

扩大读取范围前先说明触发原因；没有矛盾就停在当前级别。大文件优先使用 `tools/query_material.py` 定点查询，不直接打开全量附录、CSV 或整部原文。

## 任务路由

- 项目概览：只读接手卡；用户明确要结构/进度时，再读 README 对应小节。
- 原作分析/大纲修订：读取原作规则的相关条款、目标索引、目标大纲、目标 state 和目标原文；不要默认读取整部作品。
- state/连续性维护：先取 state 快速视图；只有跨篇或冲突核查才取完整 state。
- 结构/数量/完整性审计：只在用户明确要求或本次修改确实依赖时执行。
- 续接历史任务：先读接手卡；确需追溯时才读任务日志相关历史区段。
- Git 交付：只在用户明确要求提交时读取项目执行规则第10节和 CHANGELOG 最新记录；用户说“提交”即视为同时授权提交、推送工作分支、合并 `main` 并推送 `main`，不重复确认操作意图。

## 交付边界

不确定的原作事实必须回原文核对，仍无法确认时标记存疑并上报；发现问题不得自动扩大任务范围。子 Agent 只回报结论、文件、存疑和下一步，不复制整段原文或完整产物。

用户明确要求“提交项目”“提交并推送”或“提交”时，直接执行新建 `codex/YYYYMMDD-XX` 分支、更新 README/CHANGELOG、提交、推送工作分支、合并回 `main` 并推送 `main` 的完整流程，不二次确认远程目标或操作意图。只有认证、网络、权限或工具失败时，才停止并报告实际失败原因。未要求提交时，不主动执行 Git 提交或推送。
